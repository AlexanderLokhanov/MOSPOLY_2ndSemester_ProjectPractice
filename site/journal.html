Апрель 2025 · Этап 1
Готово Старт проекта

Постановка задачи, выбор стека и MVP

Первый месяц работы над проектом. Определена концепция: администратор запускает опрос через Telegram, бот рассылает вопрос всем зарегистрированным пользователям, собирает ответы и сохраняет причины отсутствия. Исследованы существующие решения и сформулированы функциональные требования.

Выбран стек: Python + aiogram (asyncio) + aiosqlite + SQLite. Создан репозиторий на GitHub, настроена структура папок и написан первоначальный README. Реализован proof-of-concept: бот регистрирует пользователя при первом сообщении и отвечает на /start.

Ключевые решения этапа:

  • Выбран aiogram за встроенную поддержку FSM — нужна для ветки «Причина отсутствия»
  • Права администратора проверяются по ADMIN_ID из конфига — без отдельной таблицы ролей
  • Решено использовать SQLite вместо PostgreSQL — для простоты деплоя
  • Две таблицы БД: users и attendance — этого достаточно для задачи
Май 2025 · Этап 2
Готово Ядро системы

База данных, рассыльщик и FSM

Самый технически насыщенный этап. Спроектирована схема базы данных с двумя таблицами и реализованы все операции через aiosqlite для неблокирующей работы в асинхронном окружении.

Реализован модуль broadcaster.py: при команде /poll от администратора бот выбирает все chat_id из таблицы users и последовательно рассылает сообщение с инлайн-кнопками «Да ✅» / «Нет ❌».

Проблема, с которой столкнулись: при нажатии «Нет» нужно дождаться следующего произвольного сообщения от пользователя и распознать его как причину. Решено через FSM-состояние WaitingReason — бот переходит в него после нажатия кнопки и выходит после получения текста.

# Пример записи в таблице attendance
{"user_id": 42,
 "present": False,
 "reason": "Болею, есть справка",
 "asked_at": "2025-05-14 10:30:00",
 "marked_at": "2025-05-14 10:31:45"}
